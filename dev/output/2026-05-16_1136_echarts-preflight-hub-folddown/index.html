Before / after — Rule 7 (the load-bearing diff)

BEFORE

### Rule 7 — build flow is atomic

- The four steps of Recipe 1 (Tools 1, 2, 3, 4 in §B) are
  **non-divisible**. PRISM does not return to the user
  between Tool 1 and Tool 4. ...
- Forbidden language in any user-facing message before
  Tool 4 has completed cleanly: "next steps", "would you
  like me to", ...
- Failure handling: if any tool raises, the response to
  the user is the failure (with its diagnostic), not a
  rendered HTML preview ...

AFTER

### Rule 7 — build flow is atomic (NO DEFERRED WORK)
- (existing) the four steps are non-divisible ...
- (existing) failure handling ...
+ #### Rule 7.1 — the execution model: once you respond, you terminate
+   Pattern A (build it this turn, DEFAULT, >95% of asks)
+   Pattern B (hand control back to user, genuine blocker only)
+
+ #### Rule 7.2 — forbidden phrasing (any future-tense PRISM action)
+   8-row table: "Kicking off the build now" / "I'll send you the URL
+   once the build completes" / "Building lean V1 now" / etc., each
+   paired with WHY IT'S A LIE
+
+ #### Rule 7.3 — self-check before sending any dashboard response
+   STOP-and-rewrite gates on future-tense verbs, time-deferred
+   language, implied background process, promised follow-up
+
+ #### Rule 7.4 — the clarifying-questions trap
+   PRISM identifies design questions, asks 2-3, user says
+   "do as much as you can", PRISM defers -- two turns,
+   zero artifacts. Right move on FIRST turn: pick defensible
+   defaults, run Tools 1-4, disclose the choices.

Before / after — Rule 8 (slice discipline)

BEFORE

### Rule 8 — slice complex requests, check in between slices

A "build me a dashboard" prompt rarely lands as a single
buildable specification. PRISM does NOT attempt to fulfil
the entire ask in one build flow. The discipline:
- Slice the request ...
- Build the slice atomically ...
- Hand off the URL and ASK ...
- Wait for confirmation before the next slice. Iterate.

AFTER

### Rule 8 — one-shot the build; slice only when a real blocker
+ Default disposition: ONE-SHOT the entire dashboard.
+ PRISM has 20+ tool calls per turn -- use them strategically.
+ A 4-tab / 12-widget / 5-pull dashboard fits comfortably.
+ Slicing is a FALLBACK for genuine blockers, not the default.
+
+ + One-shot when you can
+ + Use the tool budget (~8-12 calls for a typical full build)
+ + Debug in place (§H heal lexicon)
+ + Build the slice atomically (Rule 7 still governs WITHIN)
+
+ When to actually slice (genuine blockers only):
+ - Required data source returned empty AND no fallback exists
+ - User's ask is genuinely under-specified
+ - Build hit upstream contract drift unfixable in-session
+ - User explicitly asked for incremental delivery
+
+ When you do slice, do it well: pick disambiguating slice,
+ still Tools 1-4 atomically, no pre-announced multi-slice plans

New §H.5 — browser-side telemetry beacon

Single new sub-section that documents the dashboard self-report mechanism. The beacon JS (now embedded in rendering.py's DASHBOARD_SHELL) captures errors / unhandled rejections / console.error|warn / resource 404s / page_view events and POSTs JSONL to users/{kerberos}/dashboards/{id}/console_log.jsonl.

console_log.jsonl event:
{
  "ts":          "2026-05-14T18:21:00.000Z",
  "kind":        "unhandled_rejection",
  "message":     "Cannot read properties of undefined (reading 'coord')",
  "stack":       "...",
  "url":         "/profile/dashboards/rates_monitor/",
  "ua":          "Mozilla/5.0 ...",
  "viewer":      "qubbam",        // server-stamped from GSSSO cookie
  "received_at": "..."
}

The 3-step diagnostic playbook (§H.5 + §I):

  1. refresh_status.json — did the LAST refresh succeed?
  2. console_log.jsonl — what did the user's browser see?
  3. Cross-reference to localise the bug:
    • Refresh OK + console_log shows ECharts errors → manifest/chart spec bug (§C)
    • Refresh OK + console_log shows resource_404 for a dataset → Rule 5 violation (§D)
    • Refresh FAILED + console_log empty → stale HTML; fix pipeline first (§D)
    • Refresh OK + console_log empty → probably healthy; ask user for screenshot
